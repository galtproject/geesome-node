# Content Module

## Purpose

The `content` module turns uploaded, fetched, remote, or generated bytes into user-visible GeeSome content records and serves those records back through API-safe metadata, streams, previews, restore, and deletion flows.

## Owns

- Uploads from streams, local paths, URLs, directories, and remote storage IDs.
- Content metadata creation, preview generation, public metadata, `HEAD`/stream serving, and restore flows.
- Soft-delete/tombstone retention, purge-candidate listing, and purge execution after safety checks.
- User content library lookup by content ID, storage ID, manifest ID, and owner-scoped storage identity.
- Integration with storage-object sync so physical storage metadata can be shared across multiple content rows.

## Boundaries

- A `Content` row is a user/library record, not the global identity of physical bytes. Use the `StorageObject` registry for canonical physical metadata and shared references.
- Do not hard-delete physical storage from content flows unless delete-safety and retention rules prove the storage object is no longer referenced.
- Preview and generated-output references must stay visible to delete-safety and storage-space analysis.
- Serving paths must preserve permission, deleted-state, range/header, and content-length behavior.
- Re-upload or remote-import behavior must handle shared `storageId` values without assuming global content uniqueness.
